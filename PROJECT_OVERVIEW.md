# Обзор проекта: AI-ассистент для СЭД

## Что за проект?

**AI-ассистент для систем электронного документооборота (СЭД)** — это интеллектуальная система поддержки пользователей на основе больших языковых моделей (LLM), разработанная в рамках диссертационного исследования.

### Тема диссертации
"Разработка интеллектуального ассистента для поддержки пользователей систем электронного документооборота на основе LLM"

## Смысл разработки

### Проблема
Сотрудники, особенно новички, сталкиваются с трудностями при работе в СЭД:
- Сложность навигации и поиска нужных документов
- Непонимание бизнес-процессов и статусов документов
- Необходимость изучать множество инструкций и регламентов
- Частые ошибки из-за незнания правил документооборота

### Решение
Интеллектуальный ассистент, который:
- **Отвечает на вопросы** о документообороте на естественном языке
- **Проверяет загруженный документ** на корректность заполнения обязательных реквизитов
- **Использует базу знаний**: инструкции, регламенты и эталонные шаблоны документов
- **Классифицирует тип документа** (в MVP: приказ, доверенность, договор)
- **Формирует список замечаний** по отсутствующим или некорректным реквизитам

### Научная новизна
Предложен **метод контекстно-ориентированной поддержки пользователей СЭД**, который расширяет запрос пользователя следующими данными:
1. **Контекст документа** (тип, статус, этап бизнес-процесса)
2. **Контекст пользователя** (роль, отдел, права доступа)
3. **Релевантная документация** (результаты семантического поиска по базе знаний)

Это позволяет LLM генерировать более точные и контекстные ответы, чем базовый RAG-подход.

### Ключевые возможности
✅ **Делает:**
- Консультирует по инструкциям и регламентам через RAG
- Принимает файл документа (DOCX/PDF) в чате
- Классифицирует тип документа (приказ, доверенность, договор)
- Проверяет обязательные реквизиты по шаблонам/правилам из базы знаний
- Возвращает список замечаний по заполнению документа
- Сообщает об отсутствии эталонного шаблона и рекомендует обратиться к специалисту

❌ **НЕ делает:**
- Не изменяет документы
- Не выполняет операции в СЭД (создание, подписание, отправка)
- Не заменяет функционал СЭД, а дополняет его

### Целевая аудитория
- Новые сотрудники, проходящие обучение
- Опытные пользователи, которым нужна быстрая справка
- Руководители, желающие разобраться в статусе документов

### Ценность для бизнеса
- ⚡ **Ускорение обучения** новых сотрудников
- 🎯 **Снижение количества ошибок** в документообороте
- 💰 **Сокращение времени** на поиск информации
- 📉 **Уменьшение нагрузки** на службу поддержки

## Актуализация MVP (зафиксировано 2026-03-29)

Этот блок является опорным для дальнейшей разработки и имеет приоритет над ранними черновыми формулировками.

### Базовый функционал MVP
- Ассистент отвечает на вопросы по базе знаний (RAG).
- Ассистент принимает документ и проверяет обязательные реквизиты по эталонному шаблону/правилам из базы знаний.

### Зафиксированные ограничения MVP
- **Типы документов**: приказ, доверенность, договор.
- **Форматы входных файлов**: DOCX и PDF.
- **Глубина проверки**: только обязательные реквизиты (без глубокой юридической экспертизы формулировок).
- **Формат ответа**: список замечаний.
- **Для MVP**: допустимо без точной привязки к позиции в документе (страница/абзац).
- **Если шаблон не найден**: ассистент сообщает, что эталонный шаблон не найден, и рекомендует обратиться к специалисту.

### Внешние API и контур выполнения
- Под внешними API понимаются внешние (облачные) сервисы: LLM, OCR, embeddings и т.п.
- Для MVP целевой режим — **локальный контур**, без обязательной зависимости от внешних API.
- Если входной PDF является сканом, OCR может потребоваться как отдельный модуль (предпочтительно локальный).

### Компоненты системы

#### 1. **Chat UI** (Пользовательский интерфейс)
- **Технология**: React 18 + TypeScript + Vite + TailwindCSS
- **Порт**: 5173 (dev), 80 (prod)
- **Функции**:
  - Минимальная отдельная веб-страница чата

  - Отображение списка замечаний по документу
  - Отображение источников из базы знаний (опционально)
  - Стриминг ответов через Server-Sent Events (SSE)

#### 2. **AssistantApi** (Основной бэкенд)
- **Технология**: C# ASP.NET Core 8.0
- **Порт**: 5000
- **Ключевые компоненты**:
  - `AssistantPipeline` — оркестратор обработки запросов (реализует научную новизну)
  - `PromptBuilderService` — построение промптов с контекстом
  - Pipeline Steps — модульные этапы обработки
  - Controllers — REST API endpoints

#### 3. **Ollama** (LLM движок)
- **Технология**: Ollama + qwen2.5:7b (7B параметров)
- **Порт**: 11434
- **Функции**:
  - Генерация ответов на запросы пользователей
  - Локальный inference-контур без обязательных внешних API

#### 4. **Qdrant** (Векторная БД)
- **Технология**: Qdrant 1.8.4
- **Порт**: 6333
- **Функции**:
  - Хранение эмбеддингов документов из базы знаний
  - Семантический поиск по инструкциям и регламентам

#### 5. **SED Mock** (Имитация СЭД)
- **Технология**: JSON-файлы в `/data/`
- **Содержимое**:
  - Документы (тип, статус, автор, ответственный)
  - Пользователи (роль, отдел, права)
  - Бизнес-процессы (этапы, переходы, правила)

### Pipeline обработки запроса

Текущая реализация использует общий pipeline из 6 шагов. Для нового MVP фиксируются два рабочих сценария: **Q&A по базе знаний** и **проверка документа**.

#### Целевой pipeline проверки документа (MVP)

```
1. File Intake
  ↓ прием файла DOCX/PDF

2. Text Extraction
  ↓ извлечение текста и базовых метаданных

3. Document Classification
  ↓ определение типа: приказ / доверенность / договор

4. Template Retrieval (RAG)
  ↓ поиск эталонного шаблона и правил обязательных реквизитов

5. Required Fields Validation
  ↓ сравнение документа с обязательными реквизитами

6. Remark Generation
  ↓ формирование списка замечаний
```

Если на шаге 4 шаблон не найден, возвращается стандартный ответ:
"Эталонный шаблон не найден. Обратитесь к специалисту."

#### Текущий общий pipeline (исторически)

Каждый запрос проходит через **6 последовательных шагов** (Steps):

```
1. ClassificationStep
   ↓ определяет тип запроса (4 категории)

2. ContextExtractionStep
   ↓ извлекает контекст документа + пользователя из SED Mock

3. RagSearchStep
   ↓ выполняет семантический поиск по базе знаний в Qdrant

4. PromptBuildStep
   ↓ строит расширенный промпт с контекстом (НОВИЗНА)

5. LlmGenerationStep
   ↓ генерирует ответ через Ollama

6. ExperimentLoggingStep
   ↓ логирует метрики для A/B тестирования
```

### Типы запросов: текущая и целевая модель

#### Целевая модель для MVP (приоритет)

| Тип | Описание | Пример |
|-----|----------|--------|
| **KnowledgeBaseQuery** | Вопросы по базе знаний | "Как оформить доверенность?" |
| **DocumentValidationQuery** | Проверка загруженного документа | "Проверь этот договор" |

#### Историческая модель (до сужения scope)

| Тип | Описание | Пример |
|-----|----------|--------|
| **InstructionQuery** | Вопросы по инструкциям | "Как создать доверенность?" |
| **BusinessProcessQuery** | Вопросы о бизнес-процессе | "Почему документ вернулся?" |
| **DocumentSearchQuery** | Поиск документов | "Найди договоры от января" |
| **ErrorAnalysisQuery** | Анализ ошибок | "Почему не могу подписать?" |

### Пример работы ассистента

**Запрос пользователя:**
```
"Почему мой документ вернулся?"
```

**Что происходит внутри:**

1. **Классификация**: определяется тип `BusinessProcessQuery`

2. **Извлечение контекста** из SED:
```json
{
  "document": {
    "type": "доверенность",
    "status": "возврат",
    "stage": "проверка паспортных данных"
  },
  "user": {
    "name": "Иванов И.И.",
    "role": "Сотрудник",
    "department": "Отдел продаж"
  }
}
```

3. **RAG-поиск** в базе знаний:
```
- "При отсутствии паспортных данных доверенность возвращается"
- "Документ должен содержать серию и номер паспорта"
```

4. **Построение промпта**:
```
Ты ассистент системы электронного документооборота.

Контекст документа:
Тип: доверенность
Статус: возврат
Этап: проверка паспортных данных

Информация из регламентов:
1. При отсутствии паспортных данных доверенность возвращается.
2. Документ должен содержать серию и номер паспорта.

Вопрос пользователя:
Почему мой документ вернулся?

Ответь кратко и понятно.
```

5. **Генерация ответа** LLM:
```
Ваша доверенность вернулась на этапе проверки паспортных данных.
Пожалуйста, убедитесь, что в документе указаны серия и номер
паспорта. Это обязательное требование согласно регламенту.
```

## Технологический стек

### Backend
- **Язык**: C# 11
- **Фреймворк**: ASP.NET Core 8.0
- **Архитектура**: Pipeline pattern + Dependency Injection
- **HTTP-клиенты**: HttpClient для Ollama и Qdrant
- **Логирование**: Microsoft.Extensions.Logging

### Frontend
- **Язык**: TypeScript 5.5
- **Библиотека**: React 18.3
- **Сборщик**: Vite 5.4
- **Стили**: TailwindCSS 3.4
- **Стриминг**: нативный EventSource (SSE)

### LLM & Embeddings
- **LLM**: Ollama + qwen2.5:7b
  - 7 миллиардов параметров
  - Поддержка русского языка
  - Запуск локально (CPU или GPU)
- **Embeddings**: BGE-M3 (локально)
  - Основная модель эмбеддингов для RAG в MVP
  - Подходит для многоязычного семантического поиска
  - Запуск в локальном контуре без обязательных внешних API

### Vector Database
- **БД**: Qdrant 1.8.4
- **Хранилище**: персистентный volume
- **Коллекция**: `knowledge_base`
- **API**: REST HTTP

### База знаний (RAG)
- **Формат исходников**: PDF, DOCX
- **Ingestion**: Python скрипт
  - PyPDF2 для PDF
  - python-docx для DOCX
- **Chunk size**: ~500 токенов
- **Overlap**: ~50 токенов

### Infrastructure
- **Контейнеризация**: Docker + Docker Compose
- **Оркестрация**: 5 сервисов
- **Volumes**: Qdrant storage, Ollama models
- **Health checks**: для всех критичных сервисов

### Testing & Evaluation
- **Unit-тесты**: xUnit + Moq
- **Оценка качества**: скрипты в `/experiments/`
- **Dataset**: 20 вопросов с эталонными ответами
- **Метрики**: A/B сравнение (с контекстом vs без)

## Как запустить

### Предварительные требования
- Docker Desktop (или Docker Engine + Docker Compose)
- 8 GB RAM минимум
- (Опционально) NVIDIA GPU для ускорения LLM

### Быстрый старт
```bash
# 1. Клонировать репозиторий
git clone https://github.com/keyloud/AI-Assistant-SED.git
cd AI-Assistant-SED

# 2. Запустить инфраструктуру
docker-compose up -d

# 3. Дождаться загрузки моделей Ollama (может занять 5-10 минут)
docker logs -f ollama

# 4. Загрузить модели вручную (если не загрузились автоматически)
docker exec ollama ollama pull qwen2.5:7b
docker exec ollama ollama pull bge-m3

# 5. Открыть UI в браузере
http://localhost:5173
```

### API endpoints
- **Health check**: `GET http://localhost:5000/health`
- **Chat**: `POST http://localhost:5000/api/chat`
- **Chat (SSE)**: `POST http://localhost:5000/api/chat/stream`
- **Document validation (MVP minimal)**: `POST http://localhost:5000/api/documents/validate` (multipart/form-data: `file`, optional `documentTypeHint`)
